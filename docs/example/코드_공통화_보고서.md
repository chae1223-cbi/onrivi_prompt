# 코드 중복 및 공통화 조사 보고서 v1.0

본 보고서는 뉴뉴스메일 소스 코드 전반을 조사하여, 기능이 중복 구현되어 소스 분량을 필요 이상으로 차지하거나 유지보수를 어렵게 만드는 요소를 분석하고 개선 방향을 제안합니다.

---

## 1. 주요 중복 영역 분석

### 1.1 Python 에이전트 엔진 (scripts/)
현재 수집, 요약, 비주얼, 발송 등 각 에이전트가 개별 클래스로 구현되어 있으나, 기초 설정부에서 상당한 중복이 발견됩니다.

- **DB 연결 및 경로 관리**: 모든 에이전트 파일(`intake_agent.py`, `summary_agent.py` 등)이 아래와 같은 코드를 개별적으로 가지고 있습니다.
    ```python
    def __init__(self, db_path=None):
        base_dir = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
        self.db_path = db_path if db_path else os.path.join(base_dir, "db", "NewNewsMail.db")
    
    def _connect(self):
        return sqlite3.connect(self.db_path)
    ```
- **개선 제안**: `BaseAgent` 추상 클래스를 도입하거나, `scripts/utils.py`에 DB 핸들러를 공통화하여 상속 또는 주입받는 방식으로 변경 시 약 100~200라인의 중복 제거가 가능합니다.

### 1.2 API 라우트 (dashboard/app/api/)
관리자 및 서비스용 API들이 SQLite를 다루는 방식이 일관되게 중복됩니다.

- **SQLite 보일러플레이트**: 각 `route.ts` 파일이 `better-sqlite3` 모듈 임포트, `path.resolve`를 통한 DB 경로 계산, `try-catch` 응답 처리를 독립적으로 수행합니다.
- **개선 제안**: `libs/db.ts`와 같은 공통 데이터 접근 계층(DAO)을 생성하여 DB 연결을 관리하고, 공통 응답 처리 유틸리티를 사용하면 API 파일당 소스 분량을 30~50% 수준으로 줄일 수 있습니다.

### 1.3 프론트엔드 UI 및 로직 (dashboard/app/)
- **API Fetch 패턴**: `LandingPage`(`page.tsx`) 등에서 `fetch` 호출, 로딩 상태 관리, `toast` 알림 로직이 결합되어 있어 코드가 길어지는 경향이 있습니다.
- **CSS 중복**: `globals.css`가 19KB로 매우 크며, 각 페이지 내부에서도 인라인 스타일이나 중복된 스타일 정의가 보입니다.
- **개선 제안**: 
    - Custom Hook(`useApi`) 도입을 통한 API 호출 로직 공통화.
    - `dashboard/app/ui`에 정의된 공통 컴포넌트(Button, Modal 등)의 활용도 극대화.

## 2. 리팩토링 기대 효과

1. **소스 분량 감소**: 중복 로직 제거를 통해 전체 코드 라인 수(LOC)를 약 15~20% 감축 가능.
2. **유지보수 효율**: DB 경로나 공통 규격 변경 시, 여러 파일을 수정할 필요 없이 공통 모듈만 수정하여 반영 가능.
3. **안정성 향상**: 검증된 공통 에러 핸들링 로직을 프로젝트 전반에 일관되게 적용.
