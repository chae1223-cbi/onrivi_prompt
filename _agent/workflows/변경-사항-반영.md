# 변경 사항 반영 워크플로우 (Incremental Change Management)

이미 구축된 시스템에 새로운 기능 추가나 기존 요건 변경이 요청되었을 때, 전체 시스템의 안정성을 유지하며 해당 증분(Delta)만 효율적으로 반영합니다.

## 실행 단계 (Execution Steps)

1. **요구사항 등록:** 
   - 사용자의 새로운 요청을 분석하여 `docs/result/요구사항추적표.md`에 새로운 ID(예: REQ-06)로 등록합니다.
2. **증분 기획 (@요구사항-정의):** 
   - 신규 요건의 핵심 가치를 정의합니다. 기존 `docs/result/요구사항-정의-결과.md`에 '변경 이력'을 추가하는 방식으로 작성합니다.
3. **증분 분석 (@분석):** 
   - 신규 요건이 기존 로직에 미치는 영향도를 분석하고 데이터 흐름의 변화를 정의합니다. 결과물은 `docs/result/분석-결과.md`에 이력을 추가합니다.
4. **증분 설계 (@설계):** 
   - 변경이 필요한 DB 스키마나 UI 컴포넌트 구조를 설계합니다. 기존 `docs/result/설계-결과.md`에 설계 변경 사항을 반영합니다.
5. **증분 구현 (@구현):** 
   - 실제 코드를 구현하거나 수정합니다. `docs/result/구현-결과.md`에 구현 상세를 기록합니다.
6. **회귀 테스트 (@테스트-배포):** 
   - 신규 기능 검증뿐만 아니라, 기존 기능이 깨지지 않았는지(Regression) 확인합니다. `docs/result/테스트-배포-결과.md`에 결과를 기록합니다.
7. **추적표 업데이트:** 
   - 모든 단계가 완료되면 `docs/result/요구사항추적표.md`의 상태를 ✅로 업데이트합니다.

## 주의 사항 (Tips)
- 에이전트는 각 단계에서 "기존 결과물과의 차이"에 집중해야 합니다.
- 모든 문서 업데이트는 '덮어쓰기'가 아닌 'History 추가' 또는 '내용 보강' 방식을 권장합니다.
